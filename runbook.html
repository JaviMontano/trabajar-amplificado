Jarvis OS in Claude · Runbook

The operational guide for installing and using Jarvis OS

This runbook turns the playbook into action. It explains how to install the folder structure, configure Claude Cowork, create the core markdown files, run the prompts, activate cadences and audit the system.

Operating rule. Use prompts inside Claude Cowork. Cowork analyzes the request, asks for missing parameters, reviews useful attachments, proposes file changes and waits for confirmation before writing critical records.
Foundations

Before you execute

Workspace

Create one local folder for the system, authorize it in Claude Cowork, and keep the canonical files as plain text.

Two core files

CLAUDE.md defines rules and routing. MEMORY.md stores confirmed facts, decisions and preferences.

Human confirmation

Claude may propose folders, files and records. You confirm what gets created, changed or remembered.

How to use every step

Read the decision first

Each step contains a configuration decision. Understand the decision before copying the prompt.

Fill parameters

When a prompt shows inputs, replace them or ask Cowork to interview you until the parameters are clear.

Attach evidence

For CV, role, project, cadence and audit prompts, attach or paste relevant annexes: notes, job descriptions, goals, feedback, metrics or prior records.

Confirm before writing

The safe default is propose → review → confirm → write. No critical file should be overwritten silently.

F

Fundamentos · 5 conceptos antes de empezar

Estos cinco conceptos evitan errores antes del primer step. No necesitas dominar la metodología para empezar; necesitas saber qué archivo manda, qué memoria se conserva, qué límites protegen tus datos y cómo confirmar cada cambio antes de avanzar.

①Claude Cowork

Superficie de trabajo donde vives el setting con prompts guiados, anexos, archivos locales y confirmación humana. Cowork ayuda a operar contexto; no decide por ti ni debe escribir información crítica sin aprobación.

②CLAUDE.md

Archivo de texto plano que Cowork lee al inicio de CADA sesión. Contiene reglas del juego: rol, convenciones, routing, prudencia. Mantenlo bajo 300 líneas · cada línea es costo recurrente de tokens.

③MEMORY.md

Archivo plano paralelo donde Cowork escribe lo que le pides recordar entre sesiones. Crece solo. Aprendizajes, preferencias, projects activos.

④Apilamiento de reglas

Cuando pides "redacta correo a cliente X", Cowork lee en cascada: voice-principles raíz → CLAUDE.md de Correos → CLAUDE.md del Project del cliente. Resultado: correo que suena como tú y respeta convenciones de canal y de cuenta.

⑤Soberanía Digital

Tu contexto vive en archivos legibles: reglas, memoria, proyectos, tareas, plantillas y decisiones. Si cambias de herramienta, esos archivos siguen siendo tuyos. Tu forma de trabajar viaja contigo.

Cómo se conecta con el Playbook. El Playbook explica el qué, por qué y para qué. Este Runbook convierte esas decisiones en steps: qué preparar, qué adjuntar, qué prompt usar, qué archivo queda y cómo validar que no se escribió nada falso, sensible o incompleto.

Doble modo · funciona con o sin plan pago

Cada prompt está escrito para operar en dos modos. Si tu Claude Cowork tiene acceso al filesystem (plan pago + carpeta autorizada), Claude analiza el pedido, te ayuda a diligenciar los parámetros, revisa los anexos que pegues o adjuntes y crea las carpetas y archivos directamente en tu disco con confirmación. Si no tiene filesystem, el mismo prompt te entrega los mkdir exactos y el contenido literal de cada archivo para que pegues a mano. Resultado idéntico. Solo cambia quién aprieta enter.

Cómo leer cada prompt. Mira la etiqueta PROMPT · 2i sobre cada botón: el número indica cuántos inputs (0, 1, 2 o 3) tienes que rellenar. Inputs vacíos significa que solo pegas y ejecutas. En la práctica, el setting se vive desde Claude Cowork: pegas el prompt, Cowork analiza qué parámetros faltan, te pregunta lo necesario, pide anexos cuando aporten contexto y propone el archivo final antes de registrarlo. Cero edición de archivos a mano salvo en Step 0 que es chicken-and-egg (la primera carpeta debe existir antes de autorizar Cowork).

Por qué prompts de alto rendimiento

Un prompt de alto rendimiento tiene cuatro propiedades verificables: declara explícitamente sus inputs (no asume contexto), entrega instrucciones de ejecución secuenciales (no abre debates), tiene criterio de éxito explícito (Claude sabe cuándo terminó) y maneja el fallo graceful (si falta data, la marca [PENDIENTE]). Los 34 prompts de esta guía cumplen los cuatro. Si alguno falla en tu setup, dímelo y lo afinamos.

Anatomía SPEC · estilo MetodologIA

Todos los prompts de esta guía adoptan la convención canónica de MetodologIA: variables en {[snake_case]}, estructura SPEC fija (Rol · Situación · Pedido · Ejecución · Criterios), tags de procedencia inline y estándar de calidad 100 ✓. La regla en una frase: el prompt debe defender solo · si necesitas 5 idas y vueltas, el SPEC está roto.

Los 5 pilares · qué hace alto-rendimiento un prompt

①Claridad

Cero ambigüedad sobre qué se pide. Verbo imperativo + objeto + propósito + audiencia. Si Claude duda, perdió.

②Contexto

Quién pide, dónde opera, qué restricciones aplican. Lo que parece obvio para ti es invisible para Claude.

③Especificidad

Inputs tipados con {[snake_case]}. Constraints numéricos donde caben. Cero "lo razonable", siempre el dato.

④Formato

Output estructurado declarado: tabla · lista · prosa · artifact. Sección por sección si es largo.

⑤Ejemplos

Few-shot inline para los prompts críticos. Un ejemplo bueno vale más que 10 líneas de instrucciones.

B

Prerrequisitos

Antes del Step 0, valida que tienes lo mínimo: Claude Desktop instalado, plan Pro o superior activo, una carpeta de trabajo dedicada en tu disco. Tres checks de un minuto. Sin esto, los pasos siguientes fallan en silencio.

  • Cuenta Claude (plan que incluya Cowork · ver tu plan en claude.ai/settings).
  • Claude Desktop instalado en macOS o Windows. Si operas en Linux, usa el modo manual/web y valida disponibilidad antes de depender de archivos locales.
  • Permiso para crear carpetas en tu equipo (sin restricción corporativa que bloquee filesystem local).
  • Editor de texto plano · cualquiera sirve. Recomendado: Obsidian (gratis, abre la carpeta como vault).
  • 30 minutos sin interrupciones para el Step 0 · Pre-flight.
  • [Opcional] Cuenta Gmail si quieres extraer voice principles automáticamente desde correos enviados.

Aviso de privacidad. Antes de conectar Gmail, Calendar, Drive o cualquier sistema corporativo, valida con Compliance/Legal de tu empresa. Esta guía no sustituye el juicio profesional sobre datos sensibles.

Qué NO hacer en los primeros 60 minutos. No instalar plugins externos · no activar 5 conectores MCP a la vez · no escribir un CLAUDE.md de 150 líneas · no crear las 8 estaciones · no abrir 4 proyectos en paralelo. El kit es mínimo deliberado · cada cosa de más es energía gastada antes de ver retorno real. Lo que el sistema te pida en la sesión 2 se añade en la sesión 2.

Cuando termines los 60 minutos con outcome cumplido, sigue al catálogo completo · los 13 steps base son el camino largo (con 5 steps avanzados opcionales para cuando el portafolio crezca), el kit fue el atajo para empezar a ver retorno hoy. Recomendación operativa · ejecutar el catálogo a ritmo de 4h por semana durante las dos siguientes (Plan Express del Playbook §15) o adoptar el ritmo Lento de 3 meses si tu agenda lo pide (§15b del Playbook).

Auditoría · 15 min al mes

La diferencia entre un Jarvis productivo y un Jarvis de demostración está en este ritual. Quince minutos el primer lunes del mes para revisar qué Projects mueren, qué skills no se invocan y qué conectores siguen vivos sin justificación. Lo que no se audita, se ensucia.

  • ¿La memoria persistente refleja mi situación profesional actual o quedó desactualizada?
  • ¿Hay Projects sin uso que conviene archivar?
  • ¿Algún conector lleva más de 60 días autorizado sin usarse?
  • ¿Alguna scheduled task está produciendo output que ya no leo o no me sirve?
  • ¿Alguna skill propia falló en las últimas semanas y necesita revisión?
  • ¿Apareció alguna capacidad nueva en mi plan que aún no exploré y podría darme leverage?

Regla mental para tocar configuración. Cada nueva capacidad activada = nueva superficie de error y permisos a vigilar. Activa solo lo que tu próximo flujo concreto requiere y revoca lo que no usas. Vale más un Jarvis pequeño y vivo que uno grande y en desuso.

Frontera de fuentes. Esta línea conserva su propio marco Claude/Jarvis. La investigación oficial OpenAI/Codex se aplicó solo a los homólogos Codex para evitar mezclar capacidades entre plataformas.

Convención de sync · plugin externo

Markdown es canon · plugin de productividad externo es espejo opcional. Cuando se autentique un plugin · convención abstracta:

  • ID inline: sufijo [PROVIDER-ID] al final de cada línea de tarea en TAREAS.md. Permite búsqueda cruzada.
  • Sync manual en WBR: nunca bidireccional automático (evita drift + 2 fuentes de verdad).
  • Markdown gana en conflicto: si plugin y markdown disienten, el repo es la fuente.
  • Tareas creadas en plugin primero (e.g. desde mobile) se importan a TAREAS.md en el siguiente WBR.

Trigger para autenticar plugin (cuándo aplica):

  • # tareas activas cross-portafolio > 20 sostenidas ≥ 4 semanas, o
  • Aparición de colaboración multi-persona que necesita herramienta común.

Hasta que el trigger se cumpla · markdown solo es suficiente. El plugin se elige por afinidad del lector (su equipo · su flujo · su preferencia visual). El ecosistema cambia ~18 meses · cualquier nombre concreto envejece.
